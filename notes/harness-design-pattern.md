# Harness 设计模式：用 Prompt 约束驾驭 AI 行为

> 关联教程：explorer/09-skill-prompt-engineering.md、explorer/08-local-resource-and-experience.md

## 问题

如何理解 Web Access 项目中体现的 "Harness the Generative AI"（驾驭生成式 AI）设计思想？它与传统的 Workflow 编排方式有何本质区别？

## 分析

### 核心观点：两个关键机制

Web Access 体现了 Harness 设计的两个核心机制：

1. **Skills 约束**（prompt engineering 驾驭 AI 行为）：不给 AI 固定 workflow，而是通过 SKILL.md 中的技术事实引导 AI 自主决策
2. **经验复用**（沉淀-匹配循环）：跑通的内容沉淀为 site-patterns 知识文件，下次通过 match-site.mjs 脚本自动匹配

这不是简单的"给 AI 写个 prompt"，而是一套系统性的设计模式——用约束代替控制，用经验代替硬编码。

### Skills 约束的具体体现

SKILL.md 采用 Tradeoff-based 决策模式，而非传统的指令式命令。具体体现在三个层面：

**1. 三层通道调度——列出事实，让 AI 自选**

不硬编码"先搜索再抓取"的固定流程，而是列出 WebSearch / curl / CDP 各自的适用场景和 tradeoff，让 AI 根据当前任务自主选择最优通道：

| 通道 | 适用场景 | Tradeoff |
|------|----------|----------|
| WebSearch | 需要搜索引擎结果 | 信息聚合，但可能不完整 |
| curl | 静态页面、API 接口 | 快速轻量，但无法处理 JS 渲染 |
| CDP（浏览器） | 动态页面、需交互 | 功能最全，但资源消耗最大 |

这种设计的精妙之处在于：AI 不需要理解复杂的调度算法，只需要理解"什么场景适合什么工具"这个人类也能理解的事实。

**2. 浏览哲学——自然语言约束引导行为**

用自然语言描述行为约束，而非代码级的流程控制。例如：

- "先了解页面结构，再决定下一步动作"——引导 AI 先观察后行动
- "优先使用轻量级方案"——引导 AI 在满足需求的前提下选择高效路径
- "避免不必要的浏览器操作"——约束资源消耗

这些约束不是 if-else 规则，而是行为指导原则。AI 可以根据实际情况灵活执行，而非死板地走固定路径。

**3. 站点经验——约束措辞而非代码逻辑**

站点经验的注入和沉淀使用的是自然语言约束措辞：

- "必须读取"先验知识——强制但灵活，AI 可以决定如何使用经验
- "主动写入"经验文件——鼓励但不强制，AI 可以判断是否值得沉淀
- 失败时"回退通用模式"——兜底策略，保证系统不会因为缺乏经验而崩溃

对比代码级的 if-else，这种方式的优势在于：AI 可以理解经验的意图，而非机械执行。

### 经验复用的实现链路

经验复用通过一个轻量级的脚本 + 文件系统实现，形成完整的闭环：

```
match-site.mjs（47 行脚本）
├── 输入：用户 URL 或文本
├── 处理：
│   ├── 遍历 references/site-patterns/*.md
│   ├── 文件名 = 域名（如 juejin.cn.md）
│   ├── frontmatter 可选 aliases 字段（域名别名）
│   └── 用正则匹配用户文本是否包含域名或别名
├── 输出：匹配到的经验文件正文
└── 无匹配则静默退出（process.exit(0)）

SKILL.md 引导闭环：
├── 操作前：匹配到经验 → "必须读取"先验知识
├── 操作后：发现新经验 → "主动写入"经验文件
└── 失败时：回退通用模式 + 更新经验文件
```

这个设计的核心思想是：经验不需要人工整理成知识库，而是在使用过程中自然沉淀。每次成功的操作都可能产生新的经验文件，每次失败都可能更新已有经验。整个系统在使用中自动进化。

### 与 Workflow 方式的对比

将 Harness 模式与传统 Workflow 编排方式进行对比，可以更清晰地理解其设计优势：

| 维度 | Workflow（LangChain 等） | Harness（Web Access） |
|------|------------------------|----------------------|
| 控制方式 | 代码级 chain/pipeline | Prompt 中的技术事实 |
| 灵活性 | 固定流程，改流程需改代码 | AI 自主决策，改 prompt 即可 |
| 知识复用 | 硬编码在节点中 | 文件系统 + 脚本匹配 |
| 适应性 | 遇到新模式需开发者介入 | AI 可以自主回退通用模式 |
| 缺点 | 僵化、维护成本高 | 依赖 AI 的理解能力 |

Workflow 的核心问题是**僵化**：每个节点做什么、节点之间如何流转，都硬编码在代码中。想要调整流程，就必须修改代码、重新部署。而 Harness 模式通过 prompt 中的技术事实来引导 AI 的决策，使得系统能够灵活应对各种未见过的场景。

当然，Harness 也有其局限性：它依赖 AI 对技术事实的理解能力。如果 AI 无法正确理解 prompt 中的约束，系统的行为就会偏离预期。这也是为什么 SKILL.md 的撰写需要精心设计——每个技术事实都需要准确、无歧义。

## 关键发现

### 1. 技术事实驱动，而非指令驱动

Harness 的精髓在于**用 prompt 约束代替代码流程控制**。不是告诉 AI"你应该这样做"，而是告诉 AI"这些是已知事实，你根据情况判断"。这种模式充分发挥了 LLM 的推理能力，同时避免了过度约束导致的僵化。

### 2. 经验自动沉淀和复用

不依赖人工维护知识库，而是在使用过程中自然沉淀。47 行的 match-site.mjs 脚本 + 文件系统就实现了完整的经验管理闭环。这种极简的实现背后，是对"AI 能做什么"的深刻理解——AI 可以读懂自然语言写的经验文件，不需要结构化的知识库。

### 3. 失败回退机制是关键保障

经验不是保证，而是一种"先验概率"。当经验失效时（例如网站改版），系统不会崩溃，而是回退到通用模式继续工作。回退后还会更新经验文件，形成"失败-学习-进化"的闭环。这种设计思想类似于人类的"经验主义"——过去的经验指导行为，但当经验被证明错误时，我们不会固守，而是更新认知。

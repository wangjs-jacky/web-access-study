# SKILL.md Prompt Engineering 深度分析

> web-access 的 SKILL.md 展示了一种「不给 AI 固定流程，而是提供技术事实让它自主决策」的高阶 prompt 设计范式。

## 定位

- **给谁看**：想学习高级 Prompt Engineering 技巧、理解 web-access 决策框架设计的人
- **解决什么问题**：回答「SKILL.md 的 prompt 为什么这样写、背后的设计理念是什么、哪些技巧可以复用」
- **前置笔记**：[web-access-guide.md](web-access-guide.md)（产品认知），本篇聚焦 prompt 设计模式

## 一、整体组织结构

### 1.1 渐进式知识架构

SKILL.md 遵循「准备 → 理念 → 工具 → 进阶 → 经验」的认知顺序：

| 章节 | 行范围 | 功能 |
|------|--------|------|
| 元数据 | 1-11 | 版本管理、能力边界定义 |
| 前置检查 | 15-31 | 建立执行前置条件 |
| 浏览哲学 | 33-46 | 确立决策思维框架 |
| 联网工具选择 | 47-90 | 提供技术决策依据 |
| 浏览器 CDP 模式 | 92-187 | 深度技术实现 |
| 并行调研 | 188-209 | 高级协作策略 |
| 信息核实 | 210-224 | 质量保证机制 |
| 站点经验 | 225-249 | 知识积累体系 |
| References 索引 | 251-257 | 外部知识挂载点 |

**可复用技巧**：复杂能力文档采用「元数据 → 前置条件 → 核心理念 → 技术细节 → 高级策略 → 质量保证 → 经验积累」的递进结构。

### 1.2 元数据层设计

```yaml
name: web-access
version: "2.5.0"
description: 所有联网操作必须通过此 skill 处理，包括：搜索、网页抓取、登录后操作、网络交互等。
```

- **description** 明确界定 skill 边界，避免与其他能力重叠
- **version** 管理迭代和兼容性

## 二、Tradeoff-based 决策模式

这是 SKILL.md 最核心的设计创新——**不给 AI 固定流程，而是提供技术事实让它自主决策**。

### 2.1 「像人一样思考」（核心哲学）

```markdown
**像人一样思考，兼顾高效与适应性的完成任务。**

执行任务时不会过度依赖固有印象所规划的步骤，而是带着目标进入，
边看边判断，遇到阻碍就解决，发现内容不够就深入——全程围绕「我要达成什么」做决策。
```

**为什么这样设计**：传统 prompt 指定「步骤1→步骤2→步骤3」，但真实场景中网站结构多变、反爬策略升级，固定流程容易失效。目标导向 > 步骤导向。

### 2.2 四步决策框架

```
① 拿到请求 → 明确成功标准（什么算完成？需要什么信息？）
② 选择起点 → 根据任务性质，选最可能直达的方式验证
③ 过程校验 → 每步结果都是证据，用结果对照成功标准
④ 完成判断 → 确认达标后停止，不过度操作
```

**可复用技巧**：用「感知-决策-行动-评估」循环替代固定步骤。

### 2.3 场景-工具矩阵（替代条件分支）

不使用 if-then 规则，而是提供场景-工具对照表：

| 场景 | 工具 |
|------|------|
| 搜索摘要或关键词结果 | WebSearch |
| URL 已知，需要提取特定信息 | WebFetch |
| URL 已知，需要原始 HTML | curl |
| 非公开内容/反爬平台 | 浏览器 CDP |
| 需要登录态/交互操作 | 浏览器 CDP |

**为什么不用条件分支**：场景化描述让 AI 根据任务特征自主匹配工具，降低 prompt 长度同时提高灵活性。

## 三、三层通道调度

### 3.1 三层架构

```
发现层（WebSearch）→ 提取层（WebFetch/curl）→ 交互层（CDP）
     低成本 ←─────────────────────────────────────→ 高成本
     低可靠 ←─────────────────────────────────────→ 高可靠
```

### 3.2 升级触发条件

- 「多次搜索尝试后没有质的改进」→ 升级到一手来源
- 「已知静态层无效的平台」→ 直接跳到 CDP
- 「需要登录态、交互操作」→ 直接 CDP

### 3.3 程序化 vs GUI 的权衡

```
程序化方式（构造 URL、eval 操作 DOM）
├─ 优势：速度快、精确
└─ 风险：可能触发反爬

GUI 交互（点击按钮、填写输入框）
├─ 优势：确定性高、正常用户行为
└─ 代价：步骤多、速度慢
```

**创新设计**：GUI 交互也是程序化方式的有效探测——先 GUI 理解站点行为，再程序化规模化执行。

## 四、Fallback 和错误恢复

### 4.1 技术事实驱动（替代错误分类）

不定义「错误类型」，而是提供「技术事实」：

```markdown
- 页面中存在大量已加载但未展示的内容（轮播非当前帧、折叠区块、懒加载占位元素）
- DOM 中存在选择器不可跨越的边界（Shadow DOM、iframe）
- 平台返回的"内容不存在"可能是访问方式问题而非内容本身的问题
```

**为什么**：让 AI 从原理层面理解问题，而不是死记错误码。

### 4.2 登录态的智能判断

```
核心问题：目标内容拿到了吗？
→ 先尝试获取内容
→ 只有确认无法获取且判断登录能解决时，才提示用户
```

避免「遇到未登录就提示用户」的机械行为。

### 4.3 子 Agent 并发容错

- 每个子 Agent 独立创建后台 tab、独立操作、独立关闭
- 共享一个 Chrome + 一个 Proxy，通过不同 targetId 无竞态
- 任务相互独立、独立失败，不影响其他任务

## 五、代码示例和命令模板设计

### 5.1 注释即说明

每个命令上方注释说明功能和应用场景：

```bash
# 点击（POST body 为 CSS 选择器）— JS el.click()，简单快速，覆盖大多数场景
curl -s -X POST "http://localhost:3456/click?target=ID" -d 'button.submit'

# 真实鼠标点击 — CDP Input.dispatchMouseEvent，算用户手势，能触发文件对话框
curl -s -X POST "http://localhost:3456/clickAt?target=ID" -d 'button.upload'
```

### 5.2 对比教学

`/click` vs `/clickAt` 并排展示差异和适用场景。

### 5.3 警告嵌入

```markdown
很多网站的链接包含会话相关的参数（如 token），这些参数是正常访问所必需的。
提取 URL 时应保留完整地址，不要裁剪或省略参数。
```

## 六、站点经验的注入方式

### 6.1 双向知识流动

```
读取经验 ← check-deps.mjs 列出可用文件
         ← match-site.mjs 匹配目标网站
         ← Agent 根据任务自主加载

写入经验 → CDP 操作成功后发现新模式 → 主动写入站点经验文件
```

### 6.2 经验文件格式

```markdown
---
domain: example.com
aliases: [示例, Example]
updated: 2026-03-19
---
## 平台特征    — 静态信息
## 有效模式    — 正面经验
## 已知陷阱    — 负面经验
```

### 6.3 「提示而非规则」定位

> 经验内容标注发现日期，当作可能有效的提示而非保证。如果按经验操作失败，回退通用模式并更新经验文件。

## 七、Agent 行为约束和引导

### 7.1 负面清单

- **最小侵入**：不操作用户 tab，只在后台创建新 tab
- **自动清理**：任务结束自动关闭自己创建的 tab
- **风险提示**：强制向用户展示温馨提示

### 7.2 子 Agent 语言中立化

```markdown
避免 prompt 用词对子 Agent 行为的暗示：
- "搜索xx" → 暗示 WebSearch
- "抓取" → 暗示 CDP
改用中性词："获取"、"调研"、"了解"
```

**为什么**：过度指定步骤会剥夺子 Agent 判断空间，反而引入主 Agent 的假设错误。

### 7.3 信息核实的来源分级

| 可信度 | 来源 |
|--------|------|
| 一手 | 发布机构官网、公司官方页、原始论文 |
| 次级 | 权威媒体原创报道（需向用户声明） |
| 工具 | 搜索引擎用于「定位」而非「证明」 |

## 八、模块化和可扩展性

### 8.1 外部知识挂载点

```markdown
## References 索引
| 文件 | 何时加载 |
|------|---------|
| references/cdp-api.md | 需要 CDP API 详细参考时 |
| references/site-patterns/{domain}.md | 确定目标网站后 |
```

**设计意图**：按需加载，不将所有内容塞入主文档。

### 8.2 命令式 vs 描述式平衡

- **命令式**（强制执行）：`必须保留用户原有的 tab 不受影响`
- **描述式**（培养理解）：`像人一样思考，兼顾高效与适应性`

**关键约束**用命令式，**决策逻辑**用描述式。

## 九、可复用 Prompt Engineering 技巧清单

| 技巧 | 核心思想 | 适用场景 |
|------|---------|---------|
| Tradeoff-based 决策 | 提供技术事实，让 AI 自主决策 | 复杂多策略场景 |
| 四步决策框架 | 目标锚定→试探→校验→判断 | 任务型 prompt |
| 场景-工具矩阵 | 场景化描述替代条件分支 | 多工具选择 |
| 渐进式 fallback | 低成本→高成本，质量无改进则升级 | 容错设计 |
| 技术事实驱动 | 原理认知替代错误分类 | 错误处理 |
| 语言中立化 | 避免用词暗示特定操作方式 | 子 Agent 协作 |
| 经验双向流动 | 读取+写入的知识闭环 | 知识积累 |
| 注释即说明 | 代码示例嵌入用途说明 | API 文档 |
| 对比教学 | 相似功能并列展示差异 | 选择引导 |
| 外部索引 | 按需加载，主文档保持简洁 | 大型 prompt |

## 十、总结

SKILL.md 展示的核心设计理念：

> **通过提供「理解」而非「指令」，让 AI 像人一样思考和决策。**

这不是简单的指令集合，而是**知识体系 + 决策框架 + 约束机制**的综合体。适用于所有需要 AI 在复杂、不确定环境中进行决策的场景。

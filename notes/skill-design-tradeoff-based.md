# Skill 设计模式：「技术事实」驱动的 Tradeoff-based Prompt

> 不给 AI 固定流程，而是提供技术事实让它自主决策的高阶 prompt 范式。

## 定位

- **给谁看**：想设计高质量 Skill / System Prompt 的人
- **解决什么问题**：如何让 AI 在复杂、不确定环境中做出好的决策，而不是死板执行固定步骤
- **来源**：web-access SKILL.md 的设计哲学提炼

## 一、问题：为什么固定流程的 Prompt 会失败？

传统 prompt 写法——步骤1→步骤2→步骤3：

```markdown
1. 先用 WebSearch 搜索
2. 如果搜不到，用 WebFetch 抓取
3. 如果抓取失败，启动浏览器
```

**失败场景**：
- 小红书页面用 WebFetch 拿不到内容（反爬），但 prompt 说「先用 WebSearch」
- 搜索结果已经足够好，但 prompt 强制要求「再用 WebFetch」
- 网站结构变了，固定步骤完全不适用

**根本原因**：真实环境多变，固定步骤无法覆盖所有情况。

## 二、解法：Tradeoff-based 决策

### 2.1 核心原则

**提供理解，而非指令。** 告诉 AI 每个工具的 tradeoff（优势+限制），让它根据实时情况自主决策。

### 2.2 四步决策框架

```
① 目标锚定 — 定义「什么算完成」（不是「怎么做」）
② 轻量试探 — 从最小成本方式开始验证
③ 证据校验 — 每步结果对照目标，方向错了立即调整
④ 完成判断 — 达标即停，不过度操作
```

### 2.3 Tradeoff 矩阵（替代条件分支）

```markdown
| 场景 | 工具 | 适用条件 |
|------|------|---------|
| 搜索摘要/关键词 | WebSearch | 快速、低成本 |
| URL已知，定向提取 | WebFetch | 中等成本 |
| 反爬平台/登录态 | 浏览器 CDP | 高成本但最可靠 |
```

AI 读到这个矩阵，根据任务特征自主选择，不需要 if-else。

## 三、关键设计模式

### 3.1 场景-工具矩阵

```
❌ if (需要搜索) → 用 WebSearch
✅ 搜索摘要或关键词结果，发现信息来源 → WebSearch
```

**区别**：前者是条件判断，后者是场景描述。场景描述让 AI 理解「什么时候用」，而不是记住「什么时候该用」。

### 3.2 升级触发条件

```markdown
当多次搜索尝试后没有质的改进时，升级到更根本的获取方式。
```

**关键**：不说「搜索 3 次后升级」，而是用「质的改进」作为判断标准，让 AI 自主决定何时升级。

### 3.3 技术事实（替代错误分类）

```markdown
### 技术事实
- 页面中存在大量已加载但未展示的内容（轮播、折叠、懒加载）
- DOM 中存在选择器不可跨越的边界（Shadow DOM、iframe）
- 平台返回的"内容不存在"可能是访问方式的问题
```

**为什么不用错误码**：AI 理解原理后，能自行判断新场景。错误码只适用于已知的有限场景。

### 3.4 经验双向流动

```markdown
# 读取
确定目标网站后，读取对应站点经验获取先验知识

# 写入
CDP 操作成功后，主动写入新发现的站点经验

# 定位
经验内容标注发现日期，当作「可能有效的提示」而非「保证正确的事实」
```

### 3.5 语言中立化（子 Agent 协作）

```markdown
❌ "搜索 GitHub 的 API 限制"  → 暗示用 WebSearch
✅ "获取 GitHub 的 API 限制信息" → AI 自主判断用 WebSearch 还是 CDP
```

**原则**：主 Agent 描述「要什么」，不暗示「怎么做」。过度指定步骤会剥夺子 Agent 判断空间。

## 四、模板

### 4.1 Tradeoff-based Skill 模板

```markdown
---
name: your-skill
version: "1.0.0"
description: 一句话说明 skill 的能力边界
---

## 前置检查
- 检查项 1
- 检查项 2

## 核心理念
[一句话描述决策哲学]

**四步框架：**
① 拿到请求 → 明确成功标准
② 选择起点 → 从最小成本方式开始
③ 过程校验 → 每步结果都是证据
④ 完成判断 → 达标即停

## 工具选择

| 场景 | 工具 | tradeoff |
|------|------|---------|
| 场景A | 工具X | 优势... / 限制... |
| 场景B | 工具Y | 优势... / 限制... |

## 技术事实
- 事实1（帮助 AI 理解原理）
- 事实2（帮助 AI 做出正确判断）

## 经验积累
- 经验存储位置和格式
- 读取和写入时机
- 「提示而非规则」定位

## 约束
- 禁止行为
- 默认原则
- 风险提示
```

### 4.2 使用检查清单

- [ ] 每个 tool 都有 tradeoff 说明（优势+限制）
- [ ] 用场景描述替代 if-else 条件
- [ ] 提供技术事实而非错误码分类
- [ ] 设置升级触发条件而非固定重试次数
- [ ] 子 Agent prompt 用中性词（避免暗示手段）
- [ ] 经验标注时效性（「提示而非规则」）
- [ ] 关键约束用命令式，决策逻辑用描述式

## 五、适用场景

| 场景 | 为什么适合 |
|------|-----------|
| 多工具选择 | AI 根据场景自主匹配工具 |
| 环境多变 | 不依赖固定步骤，灵活调整 |
| 需要 fallback | 升级条件让 AI 知道何时换方式 |
| 多 Agent 协作 | 语言中立化避免过度指定 |
| 知识积累 | 经验双向流动，越用越聪明 |

## 六、反模式

| 反模式 | 正确做法 |
|--------|---------|
| 固定步骤列表 | 四步决策框架 |
| if-else 工具选择 | 场景-工具矩阵 |
| 错误码分类 | 技术事实 |
| 指定重试次数 | 「质的改进」升级条件 |
| "搜索 X" | "获取 X 的信息" |
| 经验当规则 | 经验当提示 |

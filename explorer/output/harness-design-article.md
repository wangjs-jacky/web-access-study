# 驾驭生成式 AI：不用 Workflow，用 Prompt 约束和经验复用

最近我在研究 [Web Access](https://github.com/eze-is/web-access) 这个项目时，发现了一个很有意思的设计理念。这个 Skill 有 5000+ Stars、核心代码 550 行，能让 Claude Code 拥有完整的联网能力——搜索、抓取、浏览器操作、并行分治，全都能做。但让我反复咀嚼的，不是它的技术实现有多花哨，而是它选择了一种**反直觉**的方式来驾驭 AI 的行为。

它不用 Workflow。

没有 LangChain 那种 `Chain → Agent → Tool` 的管道编排，没有 Dify 那种可视化流程图，没有 if-else 决策树。它用了两个看起来"很软"的机制：**Prompt 中的技术事实约束**和**经验文件的知识复用**。AI 在执行任务时是自主决策的，但它的决策被巧妙地引导到了正确的方向。

我把这种思路叫做 "Harness"——不是用铁链锁住 AI 的手脚，而是给它一个认知框架，让它自己走到对的地方去。

## 痛点：AI 太自由，后果很严重

先说问题。生成式 AI 最大的优势——自由度——恰恰也是工程化最大的敌人。

你让一个 Agent "帮我从掘金上提取小册全文"。理想情况下，它应该先了解掘金页面结构，然后选择最优路径提取。但现实往往是：

- 它可能上来就用 WebSearch 搜索"掘金小册全文"，在搜索引擎的二手信息里绕半天
- 它可能用 curl 直接抓取页面，结果拿到一堆空壳 HTML，因为掘金是 SPA 应用
- 它可能直接上浏览器，一个按钮一个按钮地点，完全不顾任务完全可以用 API 方式高效完成

每条路都能走，但每条路的效率差了十倍。**问题不是 AI 没能力，而是它不知道什么时候该用什么能力。**

传统的解决方案是 Workflow——硬编码一条决策链：先搜索、再抓取、最后浏览器兜底。但 Workflow 有三个致命问题：

1. **脆弱**：一旦网站改版，整条链可能断裂
2. **僵化**：遇到没预见的场景，只能报错或卡死
3. **维护成本高**：每个网站都可能需要不同的流程，组合爆炸

Web Access 走了另一条路。

## 机制一：用 Prompt 约束代替代码流程控制

Web Access 的核心文件是一个 `SKILL.md`——一段写给人看的 Markdown，但同时也是 AI 读取的"能力说明书"。它的设计哲学可以用一句话概括：

> **不给 AI 固定流程，而是提供技术事实让它自主决策。**

### 三层通道调度：一个典型的 Tradeoff 决策

Web Access 给 Agent 提供了三种联网通道：

| 通道 | 适用场景 | 优势 | 代价 |
|------|----------|------|------|
| WebSearch | 搜索摘要、发现信息来源 | 最快、最轻 | 二手信息、可能缺失 |
| WebFetch/curl | URL 已知，定向提取 | 直接、精确 | 无法处理 JS 渲染和登录 |
| CDP 浏览器 | 需要登录/JS 渲染/交互 | 最强、最可靠 | 最慢、最耗资源 |

注意，SKILL.md 没有写"先搜索再抓取最后上浏览器"。它做的是把每种工具的适用场景、优势和代价**原原本本地摆在 AI 面前**，让 AI 根据具体任务自主选择。

这就是 Harness 的精髓——不是告诉 AI "走哪条路"，而是告诉它"每条路通向哪里、沿途有什么"，让它自己判断。

### 四步决策框架：教 AI 怎么想

更精妙的是，SKILL.md 甚至没有直接说"选工具"这件事。它定义了一个抽象的思考框架：

```
① 拿到请求 → 明确成功标准（什么算完成？需要什么信息？）
② 选择起点 → 根据任务性质，选最可能直达的方式验证
③ 过程校验 → 每步结果都是证据，用结果对照成功标准
④ 完成判断 → 确认达标后停止，不过度操作
```

这四步完全没提"搜索""抓取""浏览器"这些具体工具。但当 AI 按照"选最可能直达的方式验证"去思考时，它自然会问自己：这个任务用搜索能直达吗？用抓取呢？用浏览器呢？——工具选择的决策就这样自然地发生了。

### 技术事实 vs 操作步骤

为了说清楚这个区别，看一个具体的例子。

**操作手册式写法**（传统 Workflow 思维）：

```
如果目标网站是小红书、微博、知乎 → 使用 CDP 浏览器模式
如果目标 URL 已知且是静态页面 → 使用 WebFetch
如果需要搜索信息 → 使用 WebSearch
```

**技术事实式写法**（Web Access 的 Harness 思维）：

```
- 页面中存在大量已加载但未展示的内容（轮播非当前帧、折叠区块、懒加载占位元素）
- DOM 中存在选择器不可跨越的边界（Shadow DOM、iframe）
- 平台返回的"内容不存在"可能是访问方式问题而非内容本身的问题
```

操作手册告诉 AI "遇到 A 做 B"，技术事实告诉 AI "这个世界的底层原理是什么"。前者只能处理预见的场景，后者能举一反三。

举个例子：当 AI 用 curl 抓取小红书页面拿到空结果时，如果它的认知是操作手册式的（"小红书 → 用 CDP"），它能正确处理。但如果遇到一个操作手册里没写的新网站呢？而如果它的认知是技术事实式的（"空结果可能是因为 JS 渲染或登录态问题"），它能自己推理出"这个新网站和之前的小红书情况类似，应该用 CDP"。

**这就是 Harness 的力量：用理解代替记忆，用推理代替查找表。**

## 机制二：经验复用——让 AI 从自己的成功中学习

如果机制一是"教 AI 怎么想"，机制二就是"让 AI 记住自己想对了什么"。

Web Access 实现了一个轻量但精致的知识闭环：跑通的操作经验沉淀为 Markdown 文件，下次遇到相同站点时自动匹配加载。

### 两个组件的协作

**组件一：站点经验文件（`references/site-patterns/`）**

每个域名一个 `.md` 文件，记录操作该网站时积累的经验：

```markdown
---
domain: xiaohongshu.com
aliases: [小红书, 红书]
updated: 2026-03-19
---
## 平台特征
SPA 应用，内容需要 JS 渲染。搜索结果页 URL 可预测。

## 有效模式
站内搜索 URL 模式：`https://www.xiaohongshu.com/search_result?keyword=xxx`
用 CDP 模式 + 程序化方式（eval DOM 操作）效率最高。

## 已知陷阱
WebFetch 返回空壳 HTML，curl 同理。
直接构造搜索 URL 比在页面上模拟搜索框输入更快。
```

**组件二：经验匹配脚本（`match-site.mjs`）**

一个只有 47 行的 Node.js 脚本，负责从用户的请求中提取域名，在经验库中查找匹配：

```javascript
// 核心逻辑：遍历所有经验文件，正则匹配用户输入
for (const entry of fs.readdirSync(PATTERNS_DIR, { withFileTypes: true })) {
  if (!entry.isFile() || !entry.name.endsWith('.md')) continue;

  const domain = entry.name.replace(/\.md$/, '');
  const raw = fs.readFileSync(path.join(PATTERNS_DIR, entry.name), 'utf8');

  // 提取 aliases（域名别名，如"小红书"→ xiaohongshu.com）
  const aliasesLine = raw.split(/\r?\n/).find((l) => l.startsWith('aliases:')) || '';
  const aliases = aliasesLine
    .replace(/^aliases:\s*/, '')
    .replace(/^\[/, '').replace(/\]$/, '')
    .split(',').map((v) => v.trim()).filter(Boolean);

  // 构建匹配正则：domain | alias1 | alias2
  const escaped = (t) => t.replace(/[.*+?^${}()|[\]\\]/g, '\\$&');
  const pattern = [domain, ...aliases].map(escaped).join('|');
  if (!new RegExp(pattern, 'i').test(query)) continue;

  // 匹配到了 → 跳过 frontmatter，输出经验正文
  // ...输出逻辑...
}
```

这个脚本做了几件很巧妙的事：

1. **域名 + 别名双重匹配**：用户说"小红书"也能匹配到 `xiaohongshu.com`
2. **正则转义**：域名中的 `.` 不会被当通配符
3. **大小写不敏感**：`Xiaohongshu` 和 `xiaohongshu` 都能匹配
4. **Frontmatter 分离**：机器读元数据，AI 读正文，各取所需

### 知识闭环：读-用-写

SKILL.md 中关于站点经验的引导，构成了一个完整的知识闭环：

```
操作前：match-site.mjs 匹配到站点经验 → AI "必须读取"先验知识
操作中：按照经验执行，效率提升 5-10 倍
操作后：发现新模式 → AI "主动写入"经验文件
失败时：回退通用模式 + 更新经验文件（标记旧经验失效）
```

关键在于"必须读取"和"主动写入"这两个引导词。这不是硬编码的函数调用，而是 SKILL.md 中的措辞策略——用"必须"的强约束确保 AI 不忽略已有经验，用"主动"的引导鼓励 AI 在成功后沉淀新知识。

**经验复用的价值不只是"快"。** 第一次操作一个网站，AI 可能需要 20 步探索：找搜索框在哪、试各种 CSS 选择器、发现需要滚动加载更多内容、遇到反爬需要换策略……有了经验后，3 步就能完成。而且经验是跨 session 持久化的——你今天用完关掉终端，明天再打开，AI 还记得昨天在这个网站上踩过的坑。

### "提示而非规则"的定位

这里有一个很聪明的元设计：经验文件被定位为"提示"而非"规则"。

> 经验内容标注发现日期，当作可能有效的提示而非保证。如果按经验操作失败，回退通用模式并更新经验文件。

为什么这很重要？因为网站会改版。如果经验被当作"规则"，AI 就会盲目遵循过时的经验，撞了南墙也不回头。定位为"提示"，AI 就会在执行时保持校验——经验说"搜索 URL 模式有效"，AI 会先验证这个模式是否还成立，再决定是否继续。

这和机制一（技术事实驱动决策）是同一个设计哲学的延续：**宁可让 AI 多想一步，也不要让它盲从。**

## 为什么不用 Workflow？一个对比论证

看到这里，你可能会问：这些能力用 Workflow 也能实现啊？LangChain 的 Chain 能编排工具调用顺序，Dify 的流程图能画 if-else 分支，甚至自己写几段 Python 也能搞定。

是的，能实现。但代价完全不同。

### 对比一：新场景适配

**Workflow 方式**：遇到新网站 → 需要开发者在流程图中新增节点 → 测试 → 部署 → 更新

**Harness 方式**：遇到新网站 → AI 根据技术事实自主推理 → 可能踩坑 → 踩完后写入经验文件 → 下次自动匹配

前者需要人介入，后者自动完成。当你要处理的网站从 10 个变成 100 个，这个差异是线性和对数的区别。

### 对比二：异常处理

**Workflow 方式**：每个节点定义 error handler → 如果异常类型没覆盖到 → 链路中断

**Harness 方式**：AI 理解异常的底层原理（"内容为空可能是 JS 渲染问题"） → 自主推理解决方案 → 不需要预定义每种异常

这不是理论推演。在 Web Access 的 SKILL.md 中，错误处理是这样写的：

```
不定义"错误类型"，而是提供"技术事实"：
- 页面中存在大量已加载但未展示的内容
- DOM 中存在选择器不可跨越的边界（Shadow DOM、iframe）
- 平台返回的"内容不存在"可能是访问方式问题
```

AI 读了这些技术事实后，遇到任何"看起来不对劲"的情况，都能从原理层面推理原因，而不是去查一张有限的错误码表。

### 对比三：维护成本

**Workflow 方式**：网站改版 → 流程断裂 → 修改流程图 → 测试 → 部署 → 更新

**Harness 方式**：网站改版 → AI 执行时发现经验失效 → 回退通用模式 → 更新经验文件 → 自动完成

Workflow 的维护成本是 O(网站数 x 改版频率)，Harness 的维护成本是 O(1)——AI 自己处理。

当然，这不是说 Workflow 没有价值。对于**确定性极高、步骤完全固定**的场景，Workflow 的可预测性是优势。但对于 Web Access 面对的场景——海量网站、多变结构、未知异常——Harness 的灵活性和自维护能力是更好的选择。

## 一个被忽略的设计细节：用词即约束

在研究过程中，我发现了一个容易被忽略但影响深远的设计细节。

当主 Agent 把任务分发给子 Agent 时，它需要自动生成子 Agent 的 prompt。问题在于，用词的选择会隐性地限制子 Agent 的行为。

比如，用户说"调研小红书上关于 Qwen 的风评"，主 Agent 可能会把任务描述写成：

> 在小红书上**搜索** Qwen 相关信息

"搜索"这个词会把子 Agent 锚定在 WebSearch 工具上——而搜索引擎根本找不到小红书的站内内容。

如果写成：

> **获取**小红书上关于 Qwen 的风评信息

"获取"是个中性词，子 Agent 会自主判断该用什么工具——大概率直接上 CDP 浏览器。

**一个词的差异，导致了完全不同的执行路径。** SKILL.md 中专门有一段提醒模型注意这个问题，建议在子 Agent prompt 中使用"获取""调研""了解"等中性词，避免"搜索""抓取"等暗示特定工具的词。

这个洞察的本质是：**在 Harness 模式中，语言本身就是一种约束机制。** 你不需要写 if-else 来阻止 AI 用错误的工具，只需要在 prompt 中选择中性的措辞，让 AI 的判断空间不被隐性限制。

## 渐进式知识架构：SKILL.md 的组织智慧

最后值得说一说的，是 SKILL.md 整体的组织方式。它不是把所有信息平铺在一个文件里，而是遵循了一个精心设计的渐进式结构：

```
元数据（版本、能力边界定义）
  → 前置检查（环境是否就绪？）
    → 浏览哲学（怎么思考联网任务？）
      → 联网工具选择（每种工具的 tradeoff）
        → 浏览器 CDP 模式（深度技术实现）
          → 并行调研（高级协作策略）
            → 信息核实（质量保证机制）
              → 站点经验（知识积累体系）
```

这个结构遵循了人类学习新事物的自然顺序：先知道"这是什么"（元数据），再确认"能不能用"（前置检查），然后理解"怎么想"（哲学），接着学习"怎么做"（工具和实现），最后掌握"怎么做得更好"（高级策略和经验积累）。

**AI 读 SKILL.md 的方式和人读文档的方式是一样的**——从宏观到微观，从原理到实践。这不是巧合，而是有意为之的设计。按需加载的知识架构——核心决策框架在主文档，CDP API 参考和站点经验在 `references/` 目录按需加载——让 AI 在每个阶段都只关注当前需要的信息，不会被海量细节淹没。

## 总结：Harness 的设计公式

研究完 Web Access，我尝试提炼出 "Harness 模式" 的设计公式：

> **Harness = 技术事实（替代操作手册） + 知识闭环（读-用-写） + 中性语言（避免隐性约束）**

这三个要素的协作方式是：

1. **技术事实**让 AI 理解世界的运行规律，而不是记忆固定的操作步骤
2. **知识闭环**让 AI 从自己的成功中学习，经验自动积累、自动复用
3. **中性语言**避免 prompt 用词对 AI 行为的隐性限制，保持决策空间开放

这三者共同构成了一种"软约束"——不是锁住 AI 的手脚，而是给它一张地图和一个指南针。AI 仍然是自主探索的，但它的探索方向被自然地引导到了正确的路上。

这个设计思路不仅适用于 Web Access 的联网场景。任何需要 AI 在复杂、不确定环境中进行自主决策的场景——数据分析、代码生成、项目管理——都可以借鉴这个模式：

- 用 tradeoff 描述替代 if-else 规则
- 用经验文件替代硬编码配置
- 用中性措辞替代隐性约束

Web Access 证明了：550 行代码 + 一份精心设计的 SKILL.md，比 5000 行的 Workflow 引擎更能驾驭 AI 的行为。不是因为代码多了不好，而是因为**驾驭生成式 AI 的正确方式，是让它理解而不是让它服从**。
